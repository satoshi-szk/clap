# CLAP API におけるプレーン値（Plain Value）の概念解説

このドキュメントは、CLAP オーディオプラグイン API のパラメータ拡張機能（`params.h`）で定義されている**「プレーン値（plain value）」**について、公式ドキュメントを補足し、より実践的な観点からその役割と本質を解説するものです。これは公式の API 定義からの引用やサマリーではなく、鈴木個人の見解を含む二次的な補足解説です。

## 1. CLAP API 定義におけるプレーン値の扱い

- プレーン値は `double` として扱われ、`clap_param_info_t` の範囲情報もこれを指す。
- オートメーション保存対象としてプレーン値を推奨。
- 自動化は「プレーン値」か「ノブ位置」の二択で、前者が範囲変更に強い。
- 単位・スケールは仕様で固定されず、プラグイン実装に委ねられる。

本ドキュメントはこの前提を踏まえた実務向け補足です。

## 2. CLAP プラグイン開発におけるプレーン値の実務上の扱い

おそらくこの 「プレーン値」という概念は、VST3 の規格から取り入れられた表現と思われます。しかし、CLAP と VST3 の仕様を比較するすると、VST3 とは異なる扱い方が必要であることが見えてきます。

CLAP の仕様には、プラグインがホストに対し、パラメータのカーブ特性（例：「このパラメータは対数スケールで表示すべき」など）を伝えるための標準的な API がありません。この制約により、ホストはプラグインから提供される未知のパラメータに対して、唯一安全な選択肢である**単純な線形（リニア）なノブやスライダー**を汎用 UI として提供せざるを得ません。

また、 VST3 における「ノーマライズ値」の仕様も存在しないため、周波数（Hz）のような知覚的に対数スケールを持つパラメータを、生の物理単位（リニアな Hz）のままプレーン値として公開すると、このホストの線形 UI では非常に操作しにくくなるという問題が発生します。

この問題を解決するため、実務では**「ホストの線形 UI が、知覚的にも線形（直感的）に感じられるような値」**をプレーン値として定義するのが妥当と考えられます。

以下の表は、プラグイン内部で実際にオーディオ処理に使われる値（真のモデル）、ホストとの通信に使われるプレーン値（DTO）、そして UI に表示されるテキスト表現の関係性を示したものです（いずれも実装選択の一例であり、仕様で固定されるものではありません）。

| パラメータの性質       | 真のモデル (DSP での内部値) | プレーン値 (ホストとの通信用 DTO)         | テキスト表現 (UI での表示) | 理由                                                 |
| :--------------------- | :-------------------------- | :---------------------------------------- | :------------------------- | :--------------------------------------------------- |
| **周波数 (Frequency)** | `440.0f` (物理単位 `Hz`)    | `log2(440.0)` (対数スケール値/オクターブ) | `"440 Hz"`                 | ホストの線形 UI がオクターブ単位で自然に振る舞うため |
| **ゲイン (Gain)**      | `0.5f` (線形振幅 `0..1`)    | `-6.02` (知覚単位 `dB`)                   | `"-6.0 dB"`                | ホストの線形 UI が dB として自然に振る舞うため       |
| **パン (Pan)**         | `-1.0f` (L) .. `1.0f` (R)   | `-1.0` .. `1.0` (線形値)                  | `"100% L"` .. `"100% R"`   | ホストの線形 UI がそのまま直感的に使えるため         |

## 3. プレーン値の正体：ホスト向け DTO

開発者が実装時に向き合うデータの流れは、多くの場合以下のようになると考えられます。

1.  **内部実装における真のモデル (True Model)**

    - プラグインの DSP（オーディオ処理）コード内で直接扱う、純粋な物理単位の値。開発者にとっての真の「モデル」はこれです。
    - 例： `float frequency_hz = 440.0f;`

2.  **プレーン値（DTO, Plain Value）**

    - ホスト・プラグインの境界を越える`double`値。ホストのオートメーションに記録される値。
    - 例： `double plain_value = log2(frequency_hz);` のように、実装者が採用する表現で DTO を定義します。

3.  **ホストの UI 表示**
    - ホストが提供する、汎用の線形ノブやスライダー、オートメーションの編集 UI は、プラグインから受け取ったプレーン値を、リニアなスケールで表示します。
    - テキスト表現が必要な場合は、プラグインが提供する `value_to_text` 関数を使用して、プレーン値をテキスト値に変換します。
    - 真のモデルが Hz でテキスト表現も Hz のようなケースでも、プレーン値（オクターブ表現）を経由して二重変換をする状況も発生します。これは仕様上避けられないと言えます。

この観点から見ると、プレーン値は純粋な「モデル」をそのまま利用するのではなく、ホストという外部境界を越えるための**DTO**として別途デザインすべきと考えられます。

---

## 結論

CLAP の仕様はプレーン値の詳細な形式や単位を固定してはいませんが、オートメーションの保存対象として「プレーン値」を推奨し、`clap_param_info_t` でもプレーン値としての最小/最大/デフォルトを扱います。実務では、これを**ホスト向け DTO**として理解し、DSP 内部値とは分離して扱うのが有効です。
